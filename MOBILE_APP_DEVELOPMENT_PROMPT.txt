================================================================================
FULFILLMATE MOBILE APP DEVELOPMENT PROMPT
================================================================================

This document provides comprehensive guidance for building the FulfillMate mobile
application for both Merchant Admins and Customers, with role-based features and
real-time synchronization with the backend and dashboards.

================================================================================
PROJECT OVERVIEW
================================================================================

FulfillMate Mobile App is a cross-platform application that serves two distinct
user types with different feature sets:

1. **MERCHANT ADMIN APP** - For merchant company admins
   - Primary use: Scan QR tickets and validate orders in real-time
   - Secondary: View orders, update order status, manage tickets

2. **CUSTOMER APP** - For end customers
   - Primary use: Track order status, receive push notifications
   - Secondary: View order history, ticket QR codes, order details

**Key Requirement:** All status changes made in the mobile app MUST reflect in
real-time on the web dashboards, and vice versa. This requires WebSocket or
polling-based real-time synchronization.

================================================================================
BACKEND ARCHITECTURE SUMMARY
================================================================================

Microservices Architecture:
- API Gateway (Port 4000) - Single entry point, API key validation
- Auth Service (Port 4001) - FulfillMate Admin authentication
- Company Service (Port 4004) - Merchant onboarding, company admin auth
- Order Service (Port 4002) - Order lifecycle management
- Ticket Service (Port 4003) - QR ticket generation and validation
- Notification Service (Port 4005) - Email dispatch via SMTP

Technology Stack:
- Node.js + TypeScript + Express
- MongoDB (separate database per service)
- Redis (rate limiting, caching)
- Kafka (event-driven communication)
- Docker Compose (orchestration)

Base URL: http://localhost:4000 (via API Gateway)

**Note:** Real-time updates currently use Kafka events. For mobile app real-time
sync, you'll need to implement WebSocket support or polling with short intervals.

AUTHENTICATION & AUTHORIZATION
================================================================================

1. MERCHANT ADMIN AUTHENTICATION:
   - Endpoint: POST /company/company-admin/login
   - Headers: your_company_api_key: <api-key>
   - Request: { companyAdminEmail, companyAdminPassword }
   - Response: { token, companyAdmin: { companyAdminName, companyAdminEmail, companyId, companyName } }
   - Token Expiry: 24 hours
   - Header: Authorization: Bearer <token>
   - Access: Only their company's orders and tickets

2. CUSTOMER AUTHENTICATION (Email-based):
   - Endpoint: POST /orders/customer (no auth required)
   - Request: { customerEmail }
   - Response: { orders: [...] }
   - Alternative: Email verification code for secure access
   - Access: Only their own orders

3. PUSH NOTIFICATION TOKENS:
   - Store device tokens (FCM for Android, APNS for iOS)
   - Endpoint: POST /notifications/register-device (to be implemented)
   - Required: deviceToken, platform (ios/android), userId, userType

REAL-TIME SYNCHRONIZATION REQUIREMENTS
================================================================================

CRITICAL: All status changes must reflect in real-time across:
- Mobile app (merchant admin)
- Mobile app (customer)
- Web dashboard (CF admin)
- Web dashboard (merchant admin)

IMPLEMENTATION OPTIONS:

Option 1: WebSocket Connection (Recommended) ⭐
- Establish WebSocket connection on app startup
- Subscribe to order/ticket update events
- Receive real-time updates when status changes
- Reconnect automatically on connection loss
- Libraries: Socket.io-client, native WebSocket

Option 2: Polling with Short Intervals
- Poll every 5-10 seconds for updates
- Check for new orders or status changes
- Less efficient but simpler to implement
- Use timestamps to only fetch new data

Option 3: Server-Sent Events (SSE)
- One-way real-time updates from server
- Simpler than WebSocket for one-way communication
- Good for status updates

RECOMMENDED APPROACH:
1. Implement WebSocket server in backend (new service or extend existing)
2. Mobile app connects via WebSocket
3. When order status changes, backend publishes to WebSocket
4. All connected clients (mobile + web) receive update instantly
5. Fallback to polling if WebSocket unavailable

WebSocket Events to Implement:
- order_status_updated - When order status changes
- order_created - When new order is created
- ticket_validated - When ticket is scanned/validated
- ticket_generated - When ticket is created
- service_status_changed - When merchant toggles service

REAL-TIME SYNCHRONIZATION ARCHITECTURE
================================================================================

BACKEND REQUIREMENTS (To Be Implemented):

1. **WebSocket Service** (New service or extend existing)
   - Port: 4007 (or integrate into API Gateway)
   - Socket.io or native WebSocket server
   - Room-based connections:
     - Merchant rooms: `company:${companyId}`
     - Customer rooms: `customer:${customerEmail}`
     - CF Admin room: `cf_admin:all`

2. **WebSocket Events**:
   ```typescript
   // Client → Server
   - join_room: Join company/customer room
   - leave_room: Leave room
   - typing: Typing indicator (for chat if implemented)

   // Server → Client
   - order_created: New order created
   - order_status_updated: Order status changed
   - ticket_validated: Ticket was validated
   - ticket_generated: New ticket generated
   - service_status_changed: Service availability changed
   ```

3. **Integration with Kafka**:
   - When Kafka event published (e.g., order_status_updated)
   - WebSocket service consumes event
   - Broadcasts to relevant rooms
   - All connected clients receive update

4. **Connection Management**:
   - Authenticate WebSocket connections
   - Handle reconnections gracefully
   - Heartbeat/ping to keep connections alive
   - Clean up disconnected clients

MOBILE APP IMPLEMENTATION:

1. **WebSocket Client Setup**:
   ```typescript
   import io from 'socket.io-client';
   
   const socket = io('ws://api.fulfillmate.com', {
     auth: { token: userToken },
     transports: ['websocket'],
     reconnection: true,
     reconnectionDelay: 1000,
   });
   ```

2. **Join Rooms**:
   ```typescript
   // Merchant Admin
   socket.emit('join_room', `company:${companyId}`);
   
   // Customer
   socket.emit('join_room', `customer:${customerEmail}`);
   ```

3. **Listen for Events**:
   ```typescript
   socket.on('order_status_updated', (data) => {
     // Update local state
     // Refresh order list
     // Show notification
   });
   ```

4. **Fallback to Polling**:
   - If WebSocket unavailable, use polling
   - Poll every 5-10 seconds
   - Check for updates since last poll
